Four layers. Clients talk to the gateway over HTTPS. Gateway routes to services via gRPC. Services own their own data stores.

Why Four Layers?

  • Client isolation: frontend teams deploy independently
  • Gateway as choke point: auth, rate limiting, logging in one place
  • Service autonomy: each service owns its schema and deploy cycle
  • Data isolation: no shared databases, no cross-service joins
The key insight: each layer has a different deployment velocity. Clients ship daily. Gateway ships weekly. Services ship on their own cadence. The layers enforce that separation.

Key Change: Connection Pooling


// v1 — new connection per request (slow)
app.get("/users/:id", async (req, res) => {
  const conn = await db.connect();      // ← expensive
  const user = await conn.query(...);
  conn.release();
  res.json(user);
});

// v2 — shared pool, checkout on demand (fast)
const pool = new Pool({ max: 20, idleTimeoutMs: 30000 });

app.get("/users/:id", async (req, res) => {
  const user = await pool.query(...);   // ← reuses connection
  res.json(user);
});
The biggest single change: connection pooling. In v1 we opened a new connection per request. In v2 we share a pool of 20 connections. This alone cut latency by 60%.

Migration Plan

  • Jan: Gateway deployed in shadow mode (reads only, no routing)
  • Feb: Auth service extracted, gateway routes auth traffic
  • Mar: Core service extracted, 10% canary
  • Apr: Full traffic migration, v1 deprecated
Four months. Shadow mode in January — the gateway reads traffic but doesn't route. February we extract auth. March we extract core at 10% canary. April full migration. Rollback plan at every stage.
